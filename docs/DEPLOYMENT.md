# デプロイガイド

このテンプレートは用途に応じて **2つの公開モード** を想定する。どちらも `index.html` +
`config.js` のビルドレス構成なので、追加のビルド工程は不要。

| モード | 内容 | 推奨ホスティング |
|---|---|---|
| **A. デモ公開** | `examples/` の架空データ、または誰に見られても問題ない旅程 | GitHub Pages（無料・シンプル） |
| **B. 実データ限定公開** | 実際の予約・宿名・便名・家族メモなどを含む自分用のしおり | **GitHub Private リポジトリ + Cloudflare Pages + Cloudflare Access** |

判断基準はシンプル：**そのしおりに公開してよいデータしか無いか？** 無いなら B を使う。
下記の個人情報スクラブを通っていないデータを A で公開しない。

---

## 公開前・個人情報スクラブ（A を使う場合の必須ゲート）

`examples/` や公開用の `config.js` には **架空データ** のみを置く。以下の実在情報を残さない。

- 実在の便名・予約番号
- 実在の宿名（公開してよい一般的な地域名・エリア名までに留める）
- 予約ステータスの生の文言（「予約済」「手配済」など）→ デモ用の中立表現へ
- 家族/私的メモ
- 実日程（具体的な年月日・実際の行動計画）→ 架空の日付/汎用サンプルへ

CI で禁止語 grep を運用するのがおすすめ（`examples/` 配下がヒットしたら失敗させる）：

```
grep -rInE '<実際の便名や宿名などのパターン>' examples/
```

## A. デモ公開（GitHub Pages）

公開リポジトリ本体・`examples/` の公開用途。認証なし・誰でも閲覧可能が前提。

1. リポジトリ設定 → Pages → Source を `main` ブランチ / `/`(root) に設定
2. `index.html` をルート、または `examples/okinawa/` 等をそのまま静的配信
3. `https://<user>.github.io/<repo>/` で公開される
4. カスタムドメインは任意（`CNAME` ファイル追加 or Pages 設定）

**制約**: 無料プランでは公開範囲の制限ができない（全世界に公開）。実データは絶対に置かないこと。

---

## B. 実データ限定公開（GitHub Private + Cloudflare）

家族・同行者だけに見せたい実データ版のための構成。**推奨・現実的な構成**として本書はこれを標準手順とする。

### 構成の考え方
- **GitHub 側**: リポジトリを Private にするだけで、コード自体への第三者アクセスを遮断
- **Cloudflare 側**: そのリポジトリを Cloudflare Pages に接続してビルド・配信し、
  **Cloudflare Access** で「閲覧者の認証（メールOTP or パスワード）」をかける
- 二重の壁になる：①GitHubはprivateなのでコードは見えない ②配信URLもAccessでロックされる

### 手順

#### 1. GitHub 側
1. 実データ用のリポジトリ（本テンプレートから作成、または `config.js` だけ差し替え）を **Private** で作成
2. `config.js` に実際の旅程データを記述（架空化は不要、非公開なので）

#### 2. Cloudflare Pages 側
1. Cloudflare ダッシュボード → **Workers & Pages** → 「アプリケーションを作成」→ **Pages** → 「Gitに接続」
2. GitHub 連携を許可し、対象の Private リポジトリを選択
3. ビルド設定：本テンプレートはビルドレスなので
   - Build command: 空（未設定）
   - Build output directory: `/`（ルート、`index.html` がある場所）
4. デプロイすると `https://<project>.pages.dev` が発行される（この時点ではまだ誰でも見える）

#### 3. Cloudflare Access で保護
1. Cloudflare ダッシュボード → **Zero Trust** → **Access** → **Applications** → 「アプリケーションを追加」
2. タイプ: **Self-hosted** を選択
3. 保護対象ドメインに Pages のURL（`<project>.pages.dev` または独自ドメイン）を指定
4. ポリシーを作成し、認証方式を選ぶ：
   - **メールOTP（推奨）**: 許可するメールアドレス（家族・同行者）を列挙 → アクセス時にそのメール宛にワンタイムコードが届く。パスワード管理不要で最も安全
   - **共有パスワード的運用にしたい場合**: 「Bypass」ポリシー等は非推奨（認証が形骸化する）。基本はメールOTPを使う
5. 保存すると、指定ドメインへのアクセス時に Cloudflare の認証画面が挟まる

#### 4. 独自ドメイン（任意）
- Cloudflare で管理しているドメイン（または Cloudflare にネームサーバーを移したドメイン）があれば、
  Pages プロジェクトにカスタムドメインを割り当てて `trip.example.com` のような形にできる
- 独自ドメインなしでも `<project>.pages.dev` + Access だけで運用可能

### 無料枠の目安（変更されうるため利用時に公式ページで確認）
- Cloudflare Pages: 無料枠で個人利用は十分（ビルド数・帯域とも寛容）
- Cloudflare Access: 無料枠で50ユーザーまで（家族・少人数の同行者利用なら十分）

## やってはいけないこと（セキュリティ注意）

- **クライアントサイドJSだけのパスワードゲート**（`if (input === "1234")` のような実装）は
  **実質的なセキュリティにならない**。HTML/JSソースは誰でも取得でき、パスワードも中身も読める。
  実データを守る目的なら B（Cloudflare Access）を使うこと。
- 実データ版のリポジトリを誤って Public に切り替えない。Private → Public の変更は
  GitHub 上で明示的な確認ダイアログが出るが、意図せぬ公開事故の主要因なので注意する。
- `examples/` に実データを紛れ込ませない（上記の個人情報スクラブ・grep チェックを運用）。

---

## まとめ（判断フロー）

```
このしおりのデータは公開しても問題ないか？
├─ Yes（架空データ / デモ） → A. GitHub Pages（無料・シンプル）
└─ No（実予約・実宿名・家族メモ等）
    → B. GitHub Private Repo + Cloudflare Pages + Cloudflare Access（メールOTP）
```
